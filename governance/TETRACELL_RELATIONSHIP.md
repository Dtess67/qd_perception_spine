# TETRACELL_RELATIONSHIP

This is a Phase I constitutional artifact. It defines the current governed relationship between TetraCell and the `qd_perception_spine` architecture so a cold session does not have to guess whether TetraCell is source authority, precursor, archive, complement, or parallel work.

## Scope and Role in Phase I

- This document governs relationship classification, not implementation.
- It does not import TetraCell logic into `qd_perception_spine`.
- It does not claim full TetraCell reconstruction inside this repo.
- It exists to remove ambiguity about whether current `qd_perception_spine` advancement depends on TetraCell.

This document is subordinate to and interpreted with:

- `governance/QD_END_TO_END_BUILD_PLAN_v0_3.md`

## Decision

**Current decision: TetraCell is a parallel architecture, not a governing dependency of `qd_perception_spine`.**

More precisely:

- TetraCell is **not** the constitutional source authority for the current perception spine.
- TetraCell is **not** the required precursor that must be completed before `qd_perception_spine` can lawfully advance.
- TetraCell is **not** merely dead archive or discarded history.
- TetraCell is a **parallel architectural line** that may contain useful structural ideas, tests, or future complement value, but those elements do not enter canonical `qd_perception_spine` by association alone.

## What This Means

### 1. Not precursor-authority
`qd_perception_spine` does not need TetraCell completion, validation, or reinterpretation in order to continue Phase I constitutional closure.

No current ratified artifact in this repo makes TetraCell a required upstream dependency for:
- empirical inheritance,
- rebuild rationale,
- constraints,
- advancement rules,
- current phase closure.

### 2. Not archive-only
TetraCell is not being classified as irrelevant, disproven, or erased.

This document does **not** say TetraCell has no value.  
It says only that value is not the same thing as current governing authority.

### 3. Parallel with possible future complement value
TetraCell may later contribute:
- structural analogies,
- detector ideas,
- decision-geometry insights,
- comparison surfaces,
- alternative or supporting architectural experiments.

But such contribution must be imported explicitly and must earn its way into canonical state under the same advancement rules as anything else.

## Why This Is the Honest Current Classification

This repo’s current ratified constitutional chain is built around:
- empirical inheritance from prior experiments,
- rebuild rationale bounded by recoverable source,
- explicit constraints,
- explicit advancement law.

Within that repo-visible chain, TetraCell is not presently named as a binding source of current perception-spine authority.

That means two dishonest extremes must both be avoided:

### Dishonest extreme 1
“TetraCell already governs this architecture implicitly.”

This would overclaim lineage not currently made explicit in ratified governance.

### Dishonest extreme 2
“TetraCell does not matter.”

This would erase a live architectural thread without governed evaluation.

The honest position is the middle one:
**parallel now, importable later only by explicit earned integration.**

## Current Boundary Rules

### B1. No implicit inheritance
No TetraCell concept, rule, geometry, or terminology becomes canonical in `qd_perception_spine` merely because it is familiar, elegant, or previously discussed.

### B2. No dependency inflation
A session must not claim that `qd_perception_spine` advancement is blocked on unresolved TetraCell questions unless that dependency is made explicit through ratified constitutional change.

### B3. No archive erasure
A session must not dismiss TetraCell as irrelevant merely because it is not currently canonical here.

### B4. No silent import
If a TetraCell idea is brought into this repo as a detector, constraint, gate, metric, or architectural dependency, that import must be named and advanced under:
- `CONSTRAINTS.md`
- `ADVANCEMENT_RULES.md`
- `BUILD_LEDGER.md`

### B5. No backfilled continuity claims
A session must not claim that current `qd_perception_spine` layers already “came from TetraCell” unless that lineage is explicitly recoverable and written into governed state.

## Allowed Future Relationship Changes

This classification may change later, but only through explicit constitutional handling.

Future governed outcomes could include:

- **parallel -> complement**  
  if specific TetraCell structures are shown to provide lawful support without becoming source authority

- **parallel -> partial precursor**  
  only if recoverable evidence shows a specific current layer truly depends on a prior TetraCell concept in a way not yet written down

- **parallel -> archive/reference**  
  only if the project later determines TetraCell is historically informative but no longer architecturally live

What is **not** allowed is silent drift from one relationship class to another.

## Import Rule

For any TetraCell concept to enter canonical `qd_perception_spine`, the project must answer all of the following:

1. What exact TetraCell element is being imported?
2. Is it a metaphor, detector, metric, gate, or architectural dependency?
3. What current problem does it solve?
4. What ratified artifact allows or requires the import?
5. What would falsify the claim that it belongs?
6. What would be lost if it were absent?
7. Has the import been recorded in `BUILD_LEDGER.md` honestly?

If these are not answered, the TetraCell import remains exploratory or diagnostic only.
